# Agent Summary — Issue #7: Concurrent Two-Writer E2E Collab Tests

## What Changed

**Only file modified:** `e2e/collab.spec.ts`

Added a `test.describe('Concurrent writes')` block nested inside the existing
`'Collaboration E2E'` describe block. Five new tests were added:

### Tests Added

1. **Both users rename the same task simultaneously — final state converges**
   Both pages double-click the same task name, fill different values, and save
   concurrently via `Promise.all`. After a 3-second CRDT settle window, both pages
   must show the same final name (one of the two written values — last-write-wins).

2. **User A adds a task while user B edits an existing task — both changes appear**
   User B edits the first task's name to a unique string; user A clicks "+ Add Task"
   at the same time user B saves. Verifies that user A sees user B's edit and user B
   sees the new task bar (count check via `.task-bar`, which renders 2 rects per task).

3. **Both users drag-resize different tasks simultaneously — both duration changes propagate**
   Gets bounding boxes of `.resize-handle` elements (nth 0 on pageA, nth 1 on pageB)
   and drags each rightward by 60px concurrently via `page.mouse`. After sync, both
   pages must have an equal positive task bar count (no divergence or crash).
   Skips gracefully if either resize handle is not found.

4. **Rapid sequential edits from both users all converge**
   Five interleaved name edits (A, B, A, B, A) on the same first task. The final
   value ("Rapid Edit 5 Final from A") must appear on pageB within 8 seconds, and
   both pages must show the same text content.

5. **User A deletes a task while user B is editing it — no crash, consistent final state**
   User B opens an inline edit on the first task while user A right-clicks it and
   selects "Delete task" from the context menu. User B then tries to save (no-op
   expected). After sync, both pages must have a lower task bar count than before,
   with equal counts (convergence), and the deleted task's name must not appear on
   pageB.

## Notes

- The pre-existing TypeScript diagnostics for `process` (lines 5, 9, 10 of collab.spec.ts)
  were present before this change — they are in the original file and were not introduced
  by these edits.
- The `.task-bar` count check in test 2 uses the fact that each task renders exactly
  2 SVG `rect` elements with class `task-bar` (fill + stroke), so adding one task
  increases the count by 2.
- Summary task bars (rendered by `SummaryBar`) are counted separately from leaf task
  bars; the "+2" expectation in test 2 may need adjustment if the default new task
  is added as a child of a summary. If this flakes, the count check can be relaxed
  to `toBeGreaterThan(initialBarCount)`.

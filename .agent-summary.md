# Agent Summary — Issue #9: Input Validation and Error Feedback

## What Changed and Why

### Created: `src/utils/taskFieldValidation.ts`
Pure, framework-agnostic validation functions:
- `validateTaskName(name)` — trims whitespace, rejects empty/whitespace-only strings
- `validateDuration(value)` — rejects non-numeric input and values < 1
- `validateEndDate(startDate, endDate)` — rejects end dates before start date

### Created: `src/utils/__tests__/taskFieldValidation.test.ts`
15 unit tests covering all acceptance criteria scenarios:
- Empty name, whitespace-only name (including tabs), valid names with surrounding whitespace
- Zero duration, negative duration, non-numeric input, empty string, valid durations
- End before start, end one day before start, same-day, after start, year boundary

### Modified: `src/components/table/InlineEdit.tsx`
- Added `validate?: (value: string) => string | null` prop
- On save: text fields are trimmed before validation; if validate returns an error string, the save is blocked and the error is shown
- Error displayed as an absolutely-positioned inline span beneath the input (does not break fixed-height row layout)
- Added `aria-invalid` and `aria-describedby` attributes referencing the error span (via `useId`)
- Input gets `.inline-edit-error` CSS class when invalid
- Error clears immediately when user starts typing; Escape also clears the error

### Modified: `src/components/table/TaskRow.tsx`
- Imported the three validation functions
- Wired `validate={validateTaskName}` to the name field InlineEdit
- Wired `validate={validateDuration}` to the duration InlineEdit
- Wired `validate={v => validateEndDate(task.startDate, v)}` to the endDate InlineEdit (closure captures task.startDate)
- Tightened `handleDurationUpdate` guard from `< 0` to `< 1` to match validation
